# Bill Splitter 需求说明书

## 1. 项目背景

H女士 和 Daisy 经常会有共同消费或互相垫付的账单，例如外卖、聚餐、奶茶、出游、团购、打车、住宿、门票等。账单来源可能是外卖软件、支付宝、微信、美团、大众点评、酒店平台或手动记录。实际结算时，经常需要先看账单截图，再结合两人的实际消费情况计算各自应收金额，最后确认谁需要转给谁多少钱。

现有 `trip-splitter` 已经实现了两人账单分摊、OCR 识别和一句话计算的基础能力。新项目继续沿用这个方向，但明确首版只服务 `H女士` 和 `Daisy` 两人，不扩展多人分账。重点是把“截图识别 + 一句话计算 + 手动校正 + 结算输出”做成稳定闭环。

## 2. 项目目标

- 先创建行程，再在对应行程下记录 H女士 和 Daisy 的两人账单
- 通过 OCR 从账单截图中提取日期、总金额、场景、交易备注等字段
- 通过一句话计算提取日期、总金额、H女士应收金额、Daisy 应收金额、场景、交易备注等字段
- 一句话计算支持凑单满减、叠加券、实际付款与原价不一致的语义场景
- 一句话计算支持中文金额识别，包括 `十七块六毛八` 和 `壹佰贰拾叁元肆角伍分`
- 一句话计算支持同一人多件商品自动求和，例如 `H女士点了奶茶28和小食16`
- 一句话计算支持加减法表达式，例如 `一共花费29.9+39.9`
- 支持 OCR 识别结果与一句话计算结果组合使用：OCR 先提取账单事实，一句话再提取两人分摊
- 支持用户手动校正识别结果，避免自动识别错误影响结算
- 汇总当前行程下的总金额、H女士分摊总额、Daisy 分摊总额
- 支持复制或发送当前行程的账单分析
- 输出清晰的最终结算结果，可直接用于微信或支付宝转账

## 3. 用户与角色

- 主要用户：H女士
- 固定分摊对象：Daisy
- 付款人：H女士 或 Daisy
- 数据来源：账单截图、外卖软件截图、团购平台截图、支付宝/微信交易截图、手工输入文字

## 4. 典型场景

### 4.1 外卖账单截图

Daisy 在外卖软件下单，总金额 86.40 元。截图中包含商家、下单时间、实付金额、优惠信息。H女士 上传截图后，系统识别日期、总金额和场景“外卖”。随后 H女士 输入一句话说明：“Daisy 付款，H女士吃 32，Daisy 吃 54.4，备注：晚餐外卖”。系统自动补全两人应收金额。

### 4.2 团购餐饮

Daisy 买了双人套餐，总金额 198 元，但 H女士 额外点了 28 元饮料。用户输入：“6月8日餐饮，Daisy付款，总金额226，H女士应收127，Daisy应收99，备注：团购套餐加饮料”。系统提取字段并生成结算结果。

### 4.3 出游账单

旅行中有住宿、门票、奶茶、交通等账单。有些账单均分，有些账单按实际消费分。每笔账单保存后，系统汇总当前账本，计算 H女士 和 Daisy 最终应付或应收差额。

### 4.4 OCR + 一句话组合录入

用户先上传账单截图，OCR 自动填入日期、总金额、场景、描述。截图无法判断两人实际分摊金额，因此用户再输入一句话：“H女士 45，Daisy 38.5，Daisy 付款，备注：奶茶和小食”。系统把 OCR 字段和一句话字段合并，形成完整账单。

## 5. 核心假设

- 首版固定只有 H女士 和 Daisy 两人
- OCR 只能辅助提取账单字段，不能保证完全准确
- 一句话计算优先支持高频中文格式，不追求理解所有自然语言
- 用户愿意在保存前检查和修正字段
- 不接入支付宝、微信、外卖平台等官方 API
- 纯前端本地存储足以完成 MVP 验证

## 9. 一句话计算需求

### 9.1 支持字段

一句话计算需要尽量提取：

- `date`：日期
- `amount`：总金额
- `payer`：付款人，H女士 或 Daisy
- `shareH`：H女士应收金额
- `shareDaisy`：Daisy 应收金额
- `scene`：场景，例如外卖、餐饮、奶茶、交通、住宿、门票、团购、购物、其他
- `note`：交易备注
- `description`：账单描述

### 9.2 推荐输入格式

```text
6月8日外卖，Daisy付款，总金额86.4，H女士32，Daisy54.4，备注：晚餐外卖
```

```text
2026-06-08 餐饮 Daisy付款 实付226 H女士应收127 Daisy应收99 备注：团购套餐加饮料
```

```text
奶茶 45，H女士付款，H女士 20，Daisy 25，备注：两杯奶茶
```

### 9.3 解析规则

- 日期支持 `2026-06-08`、`2026/6/8`、`6月8日`
- 总金额优先匹配 `总金额`、`实付`、`实际付款`、`合计`、`共计`
- 金额支持阿拉伯数字、中文口语数字和中文大写数字
- 中文金额示例：`十七块六毛八` = 17.68，`壹佰贰拾叁元肆角伍分` = 123.45
- 如果未明确总金额，但 H女士和 Daisy 金额都存在，则总金额默认为两人应收金额之和
- 如果同一人出现多笔金额，系统应按该人的金额合计计算分摊
- 如果总金额字段中出现加减法表达式，系统应先计算表达式结果
- H女士金额可匹配 `H女士32`、`H女士应收32`、`H女士花32`、`我32`
- Daisy 金额可匹配 `Daisy54.4`、`Daisy应收54.4`、`daisy花54.4`
- 付款人可匹配 `Daisy付款`、`Daisy付`、`H女士付款`、`我付`
- 场景按关键词识别：外卖、餐饮、奶茶、交通、住宿、门票、团购、购物、其他
- 备注优先匹配 `备注：`、`说明：` 后面的内容
- 如果没有显式备注，但句子包含 `凑300-500买衣服活动` 这类活动短语，系统应把该短语作为交易备注
- 消费物品应从 `买衣服`、`点奶茶`、`吃火锅` 等语义中提取具体物品，例如 `买衣服活动` 提取为 `衣服`

### 9.4 计算规则

如果一句话只提供消费权重，而非实际总金额分摊：

```text
Daisy付款，daisy花260，H女士花300，实际付款360
```

则按权重换算：

```text
H女士应收 = 实际付款 * H女士权重 / (H女士权重 + Daisy权重)
Daisy应收 = 实际付款 * Daisy权重 / (H女士权重 + Daisy权重)
```

如果一句话直接提供两人应收金额，则直接使用。

### 9.5 凑单满减与叠券

当一句话出现 `凑单`、`满减`、`叠加券`、`优惠后付款`、`券后付款` 等语义时，系统应优先识别：

- Daisy 的原价、凑单金额或消费权重
- H女士 的原价、凑单金额或消费权重
- 实际付款、券后付款或优惠后付款金额

示例：

```text
为了凑单满减，Daisy女士付款260，H女士凑了一件300，实际价格560，叠加券后付款360
```

系统应计算：

```text
Daisy分摊 = 360 * 260 / 560 = 167.14
H女士分摊 = 360 * 300 / 560 = 192.86
```

也需要支持活动门槛写法：

```text
凑300-500买衣服活动，Daisy凑了260，H女士凑了300，Daisy女士一共付款360
```

系统应忽略 `300-500` 这类活动范围，不把它当作减法表达式；应把 `260` 和 `300` 作为两人的凑单权重，把 `360` 作为实际付款总金额，同时提取消费物品和交易备注：

```text
总金额 = 360.00
Daisy分摊 = 360 * 260 / (260 + 300) = 167.14
H女士分摊 = 360 * 300 / (260 + 300) = 192.86
付款人 = Daisy
消费物品 = 衣服
交易备注 = 凑300-500买衣服活动
```

### 9.6 多件商品与分别点单

当一句话描述两人分别点了多件商品时，系统应先汇总每个人的商品金额，再决定是否需要按优惠后付款比例换算。

示例：

```text
H女士点了奶茶28和小食16，Daisy点了汉堡32，Daisy付款76
```

系统应计算：

```text
H女士分摊 = 28 + 16 = 44
Daisy分摊 = 32
总金额 = 76
```

如果存在优惠：

```text
H女士点28和16，Daisy点32，满70减10，Daisy实付66
```

系统应按 44:32 分摊 66：

```text
H女士分摊 = 38.21
Daisy分摊 = 27.79
```

### 9.7 加减法表达式

当用户输入类似下面的表达式时：

```text
一共花费29.9+39.9
```

或：

```text
二十九块九加三十九块九
```

系统应自动计算：

```text
总金额 = 69.80
H女士分摊 = 34.90
Daisy分摊 = 34.90
```

识别要求：

- 支持 `+`、`-`、`加`、`减`
- 支持中文口语金额和阿拉伯数字混用，例如 `二十九块九加39.9`
- 支持输入容错：用户误输 `价` 时，按 `加` 处理，例如 `二十九块九价三十九块九`
- 如果没有写两人各自金额，则按总金额默认均分

支持 `+` 和 `-`，例如：

```text
总金额100-20+8，Daisy付款
```

应解析为：

```text
总金额 = 88.00
H女士分摊 = 44.00
Daisy分摊 = 44.00
付款人 = Daisy
```

## 10. OCR 识别需求

### 10.1 识别对象

- 外卖软件账单截图
- 团购平台订单截图
- 支付宝或微信交易截图
- 酒店、交通、门票等订单截图
- 其他包含日期和金额的账单图片

### 10.2 OCR 输出

OCR 模块需要：

- 展示上传图片预览
- 展示识别进度
- 展示 OCR 原文
- 从 OCR 原文提取结构化字段

### 10.3 OCR 字段提取

需要尽量提取：

- 日期
- 总金额
- 场景
- 描述
- 交易备注

外卖截图可优先识别：

- 商家名
- 下单时间
- 实付金额
- 商品或订单描述

如果 OCR 只能识别部分字段，也应填入已识别字段，不阻止用户继续手动补全。

### 10.4 OCR 与一句话计算联动

OCR 不负责判断 H女士 和 Daisy 的真实分摊金额。推荐流程：

1. 用户上传账单截图
2. OCR 填入日期、总金额、场景、消费物品、备注
3. 用户输入一句话说明两人分摊
4. 一句话计算补全 H女士应收、Daisy 应收、付款人
5. 用户检查后保存

## 11. 计算规则

### 11.1 单笔校验

- 总金额必须大于 0
- 付款人只能是 H女士 或 Daisy
- H女士应收和 Daisy 应收必须大于等于 0
- H女士应收 + Daisy 应收 应等于总金额
- 允许 0.01 元以内的小数误差
- 若两人应收都为空，默认均分
- 若只填一人应收，另一人自动等于总金额减去已填金额

### 11.2 汇总计算

```text
H女士净额 = H女士已付款总额 - H女士应收总额
Daisy净额 = Daisy已付款总额 - Daisy应收总额
```

- H女士净额大于 0：Daisy 需要转给 H女士
- H女士净额小于 0：H女士 需要转给 Daisy
- H女士净额等于 0：两人已平账

### 11.3 示例

```text
账单1：Daisy付款 86.40，H女士应收 32，Daisy应收 54.40
账单2：H女士付款 45，H女士应收 20，Daisy应收 25

H女士已付 45，应收 52，净额 -7
Daisy已付 86.40，应收 79.40，净额 +7

结论：H女士需转给 Daisy 7.00 元
```

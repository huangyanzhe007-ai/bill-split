# bill-split 开发操作手册

## 1. 项目目标

开发一个专门服务 `H女士` 和 `Daisy` 的两人账单分摊手机端 MVP。首版不扩展多人，用户必须先创建行程，再进入行程创建账单；把 OCR 识别、一句话计算、手动校正、行程汇总和账单分析分享做稳定。

## 2. 开发原则

### 2.1 固定两人，不做成员系统

首版角色固定：

```js
const PEOPLE = ['H女士', 'Daisy'];
```

不要做添加成员、删除成员、多人结算、多人转账建议。所有计算围绕 H女士 和 Daisy。

### 2.2 OCR 和一句话计算必须保留

账单录入有三种方式：

1. 纯手工录入
2. 一句话计算后同步到表单
3. OCR 识别账单截图后，再用一句话计算补全两人分摊金额

OCR 主要负责识别账单事实：日期、总金额、场景、消费物品、交易备注。一句话计算主要负责识别分摊事实：付款人、H女士应收、Daisy 应收，也可以补充日期、总金额、场景、备注。

### 2.3 不依赖支付平台 API

首版不接支付宝、微信、美团、饿了么、银行等 API。OCR 识别截图 + 手动确认更容易落地，也更符合个人工具的实现成本。

### 2.4 先保证计算正确

界面可以简单，但金额必须清楚：

- 总金额
- H女士应收
- Daisy 应收
- H女士已付
- Daisy 已付
- 最终谁转给谁

## 6. 核心函数设计

### 6.1 金额格式化

```js
function formatMoney(value) {
  return Number(value || 0).toFixed(2);
}
```

### 6.2 计算两人分摊

规则：

- 两人应收都为空：默认均分
- 只填写 H女士应收：Daisy 应收 = 总金额 - H女士应收
- 只填写 Daisy 应收：H女士应收 = 总金额 - Daisy 应收
- 两人都填写：校验两人应收之和是否等于总金额
- 允许 0.01 元以内误差

```js
function computeShares(amount, shareHInput, shareDaisyInput) {
  const total = Number(amount);
  const h = shareHInput === '' ? null : Number(shareHInput);
  const daisy = shareDaisyInput === '' ? null : Number(shareDaisyInput);

  if (h === null && daisy === null) {
    return { shareH: total / 2, shareDaisy: total / 2 };
  }
  if (h === null) {
    return { shareH: total - daisy, shareDaisy: daisy };
  }
  if (daisy === null) {
    return { shareH: h, shareDaisy: total - h };
  }
  return { shareH: h, shareDaisy: daisy };
}
```

### 6.3 汇总计算

```text
H女士已付 = 所有 payer 为 H女士 的账单 amount 总和
Daisy已付 = 所有 payer 为 Daisy 的账单 amount 总和

H女士应收 = 所有 shareH 总和
Daisy应收 = 所有 shareDaisy 总和

H女士净额 = H女士已付 - H女士应收
Daisy净额 = Daisy已付 - Daisy应收
```

如果 `H女士净额 < 0`，说明 H女士 需要转给 Daisy。

如果 `H女士净额 > 0`，说明 Daisy 需要转给 H女士。

## 7. 一句话计算实现

### 7.1 需要解析的字段

```js
{
  date,
  scene,
  description,
  totalAmount,
  payer,
  shareH,
  shareDaisy,
  note
}
```

### 7.2 文本标准化

处理中文标点、空格和大小写：

```js
function normalizeSentence(text) {
  return text
    .replace(/，/g, ',')
    .replace(/。/g, '.')
    .replace(/；/g, ';')
    .replace(/：/g, ':')
    .replace(/\s+/g, ' ')
    .trim();
}
```

### 7.3 日期解析

支持：

- `2026-06-08`
- `2026/6/8`
- `6月8日`

如果没有年份，默认使用当前年份。

### 7.4 场景解析

关键词：

```js
const SCENES = ['外卖', '餐饮', '奶茶', '交通', '住宿', '门票', '团购', '购物', '其他'];
```

规则：

- 文本包含关键词则使用对应场景
- 包含“饿了么”“美团外卖”“外卖订单”时归为外卖
- 包含“酒店”“民宿”时归为住宿
- 包含“打车”“滴滴”“地铁”时归为交通

### 7.5 总金额解析

优先匹配：

- `总金额86.4`
- `实付86.4`
- `实际付款86.4`
- `合计86.4`
- `共计86.4`
- `金额86.4`
- `券后付款三百六十`
- `实际付款壹佰贰拾叁元肆角伍分`

金额解析必须支持：

- 阿拉伯数字：`86.4`
- 中文口语金额：`十七块六毛八`
- 中文大写金额：`壹佰贰拾叁元肆角伍分`

如果没有总金额，但 H女士 和 Daisy 金额都识别到了，总金额等于两者之和。

### 7.6 两人金额解析

H女士：

- `H女士32`
- `H女士应收32`
- `H女士花32`
- `我32`
- `我应收32`

Daisy：

- `Daisy54.4`
- `Daisy应收54.4`
- `daisy花54.4`

注意：金额解析要避免把总金额误识别成某个人金额。

### 7.7 权重换算

如果一句话中出现：

```text
Daisy付款，daisy花260，H女士花300，实际付款360
```

这里 `260` 和 `300` 是原价或权重，不是最终应收。需要按比例换算：

```text
H女士应收 = 360 * 300 / (260 + 300)
Daisy应收 = 360 * 260 / (260 + 300)
```

建议规则：

- 若存在 `实际付款` 或 `实付`，且两人字段使用 `花`，按权重换算
- 若字段使用 `应收`，直接作为应收金额
- 如果同一人出现多笔商品金额，应先对该人的金额求和

### 7.8 凑单满减和叠券

如果一句话中出现：

```text
为了凑单满减，Daisy女士付款260，H女士凑了一件300，实际价格560，叠加券后付款360
```

这里 `260` 和 `300` 是两人的原价或凑单权重，`360` 是券后实付金额。应按比例换算：

```text
Daisy分摊 = 360 * 260 / (260 + 300) = 167.14
H女士分摊 = 360 * 300 / (260 + 300) = 192.86
```

活动门槛写法也要支持：

```text
凑300-500买衣服活动，Daisy凑了260，H女士凑了300，Daisy女士一共付款360
```

解析结果：

```text
总金额 = 360.00
Daisy分摊 = 360 * 260 / (260 + 300) = 167.14
H女士分摊 = 360 * 300 / (260 + 300) = 192.86
付款人 = Daisy
消费物品 = 衣服
交易备注 = 凑300-500买衣服活动
```

这里 `300-500` 是活动范围，不参与总金额计算；`Daisy女士一共付款360` 是最终实付，不计入 Daisy 的个人凑单权重。`买衣服` 用来提取消费物品，完整的 `凑300-500买衣服活动` 可作为交易备注。

触发关键词：

- `凑单`
- `满减`
- `叠加券`
- `优惠后付款`
- `券后付款`
- `实际付款`

### 7.9 多件商品自动求和

示例：

```text
H女士点了奶茶28和小食16，Daisy点了汉堡32，Daisy付款76
```

解析要求：

```text
H女士分摊 = 28 + 16 = 44
Daisy分摊 = 32
总金额 = 76
```

如果同时存在优惠：

```text
H女士点28和16，Daisy点32，满70减10，Daisy实付66
```

解析要求：

```text
H女士分摊 = 66 * 44 / 76 = 38.21
Daisy分摊 = 66 * 32 / 76 = 27.79
```

### 7.10 加减法表达式计算

如果一句话中出现：

```text
一共花费29.9+39.9
```

也支持：

```text
二十九块九加三十九块九
二十九块九价三十九块九
二十九块九加39.9
```

解析要求：

```text
总金额 = 69.80
H女士分摊 = 34.90
Daisy分摊 = 34.90
```

规则：

- 支持 `+`、`-`、`加`、`减`
- 支持中文口语金额和阿拉伯数字混用，例如 `二十九块九加39.9`
- 容错支持把 `价` 当作 `加` 处理，方便修正手机输入误差
- 只允许金额、加号、减号、小数点和中文金额参与计算
- 如果没有写两人各自金额，则按总金额默认均分
- 如果同时写了付款人，例如 `总金额100-20+8，Daisy付款`，则付款人也要同步到表单

### 7.11 备注解析

匹配：

- `备注: xxx`
- `说明: xxx`
- `交易备注: xxx`

备注字段保留到句末或分隔符前。

如果没有显式 `备注:`，但句子开头包含活动短语，也可以自动生成备注：

```text
凑300-500买衣服活动
```

同时从 `买衣服`、`点奶茶`、`吃火锅` 等短语中提取消费物品。

## 8. OCR 识别实现

### 8.1 技术建议

继续使用 `Tesseract.js`：

```html
<script src="https://cdn.jsdelivr.net/npm/tesseract.js@4.0.2/dist/tesseract.min.js"></script>
```

### 8.2 OCR 流程

1. 用户选择图片
2. 页面展示预览
3. 调用 `Tesseract.recognize(file, 'chi_sim+eng')`
4. 展示识别进度
5. 输出 OCR 原文
6. 提取结构化字段
7. 同步到录入表单

### 8.3 OCR 字段提取

从 OCR 原文中提取：

- 日期
- 总金额
- 场景
- 描述
- 交易备注

金额识别优先级：

1. `实付`
2. `支付金额`
3. `合计`
4. `总计`
5. `订单金额`
6. 最后才取第一个金额

外卖截图场景识别：

- 出现 `美团`、`饿了么`、`外卖`、`配送`，场景为外卖
- 出现 `商家`、`门店`，可作为描述候选
- 出现 `下单时间`、`支付时间`，可作为日期候选

### 8.4 OCR 与一句话联动规则

合并字段时建议：

- OCR 先填日期、总金额、场景、消费物品、备注
- 一句话再填付款人、H女士应收、Daisy 应收
- 一句话中明确出现日期、总金额、场景时，可以覆盖 OCR
- 用户手动修改过的字段，不要再被自动覆盖

首版可以简化：点击 OCR 按钮同步一次，点击一句话计算按钮再同步一次，用户最后手动确认。

- 金额为 0 或负数时阻止保存

### 10.4 汇总结算

- Daisy 垫付较多时，提示 H女士 转给 Daisy
- H女士 垫付较多时，提示 Daisy 转给 H女士
- 两人已平账时，提示无需补差
- 删除账单后汇总实时更新
- 刷新页面后数据保留
